# SAMVIL v3.6 Phase 4 — Domain Packs

**Date:** 2026-04-26
**Base:** v3.5.0 Telemetry + Run Observability
**Target:** v3.6.0

## Goal

Add portable Domain Packs that give SAMVIL domain-specific product, data,
interview, build, and QA context without bloating skill bodies. Phase 4 should
answer: what does this type of app normally need, what risks should we probe,
what data model is common, and what should QA verify beyond generic framework
checks?

## Boundary

Pattern Registry answers **how to build with a framework or implementation
pattern**. Domain Packs answer **what a domain tends to require**.

Examples:

- Pattern Registry: Next.js App Router, Vite React, Phaser, Expo, Recharts.
- Domain Packs: SaaS dashboard, browser game, mobile habit tracker.

## Phase 4 Outputs

| Area | Output |
|---|---|
| Domain pack schema | `references/domain-pack-schema.md` |
| Domain registry | `mcp/samvil_mcp/domain_packs.py` |
| MCP tools | `list_domain_packs`, `read_domain_pack`, `render_domain_context` |
| Skill wiring | Interview/Design/Build/QA can request domain context by solution type |
| Dogfood | synthetic and small-app checks prove domain context is useful and deterministic |

## Non-Goals

- Full marketplace or user-authored pack lifecycle.
- LLM-generated pack content.
- Replacing Pattern Registry.
- Migrating every skill body in this phase.

## Release Bar

- Full test suite green.
- Pre-commit check green.
- Domain Pack tools are deterministic and file-only.
- At least three packs provide interview, design, build, and QA context.
- Skills request domain context by tool call rather than embedding long prose.
